项目规划（Project Planning）

一、学习目标

完成本课后，学习者应能够：

  1. 理解项目规划在工程管理中的核心作用和价值。
  2. 掌握项目规划的关键步骤，包括需求分析、范围定义、里程碑设定和风险预估。
  3. 能够使用工具和方法制定可执行的项目计划。
  4. 具备识别依赖关系、分配资源和管理时间表的能力。

二、核心概念

项目规划是工程管理的第一步，是把想法和目标转化为可执行行动的过程。它是确保项目按时、按预算、高质量完成的基础。核心概念包括：

  1. 项目目标与范围

    • 项目目标（Objectives）：明确项目要解决的问题或要达成的成果。
    • 项目范围（Scope）：定义项目的边界，包括需要完成的任务和不包括的任务。

  2. 需求分析（Requirements Analysis）

    • 收集和整理利益相关者的需求。
    • 识别功能需求和非功能需求（如性能、安全性、可维护性）。

  3. 工作分解结构（WBS, Work Breakdown Structure）

    • 将项目拆解为可管理的小任务。
    • 每个任务明确负责人、时间估算和依赖关系。

  4. 时间规划与里程碑（Timeline & Milestones）

    • 制定甘特图或时间表，设定关键节点（Milestones）。
    • 里程碑用于监控项目进度和阶段性成果。

  5. 资源与依赖管理（Resource & Dependency Management）

    • 确定所需人员、设备、预算等资源。
    • 识别任务之间的依赖关系，避免瓶颈和阻塞。

  6. 风险识别与缓解（Risk Identification & Mitigation）

    • 列出潜在风险，评估发生概率和影响。
    • 制定应对措施，如备用方案或缓冲时间。

敏捷方法（Agile Methodologies）

一、学习目标

完成本课后，学习者应能够：

  1. 理解敏捷方法的核心理念和价值观。
  2. 掌握敏捷开发的常见框架（Scrum、Kanban 等）及其实操流程。
  3. 能够在团队中设计敏捷流程、制定迭代计划和管理任务。
  4. 学会通过敏捷方法提高项目透明度、交付速度和团队协作效率。

二、核心概念

敏捷方法是一种以快速迭代、持续交付和响应变化为核心的项目管理方法。其主要理念来源于《敏捷宣言》，包括以下四大价值观：

  1. 个体与互动胜过流程与工具
  2. 可工作的软件胜过详尽的文档
  3. 客户合作胜过合同谈判
  4. 响应变化胜过遵循计划

风险管理（Risk Management）

一、学习目标

完成本课后，学习者应能够：

  1. 理解风险管理在项目和工程管理中的核心作用。
  2. 掌握识别、评估、应对和监控风险的完整流程。
  3. 能够使用工具和方法制定风险缓解策略。
  4. 具备在项目执行中持续监控风险的能力。

二、核心概念

风险管理是系统地识别、评估和处理项目或组织可能遭遇的不确定事件的过程。

核心概念

  1. 风险（Risk）
    • 可能发生的事件或条件，若发生会对项目目标产生正面或负面影响。
    • 类型：技术风险、资源风险、市场风险、外部环境风险等。
  2. 风险识别（Risk Identification）
    • 系统地发现潜在风险。
    • 方法：头脑风暴、历史数据分析、专家访谈、SWOT 分析。
  3. 风险评估（Risk Assessment）
    • 概率（Likelihood）：风险发生的可能性
    • 影响（Impact）：风险发生后对项目目标的影响程度
    • 风险优先级矩阵（Probability × Impact）用于排序
  4. 风险应对策略（Risk Response）
    • 避免（Avoid）：改变计划以消除风险
    • 减轻（Mitigate）：采取措施降低风险发生概率或影响
    • 转移（Transfer）：将风险转移给第三方（如保险、外包）
    • 接受（Accept）：意识到风险存在，但不采取主动措施
  5. 风险监控（Risk Monitoring）
    • 持续跟踪风险状态
    • 更新风险登记册（Risk Register）
    • 根据风险变化调整应对措施

依赖管理（Dependency Management）

一、学习目标

完成本课后，学习者应能够：

  1. 理解依赖管理在项目执行中的重要性。
  2. 掌握识别和追踪依赖关系的方法。
  3. 能够制定依赖缓解策略，避免项目瓶颈。
  4. 熟悉工具与实践，用于持续监控依赖状态。

二、核心概念

依赖管理是识别、分析和协调项目中各任务或团队之间依赖关系的过程，确保项目顺利推进。

核心概念

  1. 依赖类型

    • 内部依赖（Internal Dependencies）：项目内部任务或团队之间的相互依赖。
    • 外部依赖（External Dependencies）：依赖外部团队、供应商或第三方资源。

  2. 依赖关系管理（Dependency Management）

    • 识别任务间依赖关系，明确关键路径。
    • 规划依赖解决方案，避免延迟蔓延。

  3. 关键路径（Critical Path）

    • 决定项目最短完成时间的任务序列。
    • 关键路径上的延迟会直接影响项目交付。

  4. 依赖缓解策略（Mitigation Strategies）

    • 调整任务顺序
    • 并行开发可行模块
    • 提前获取外部资源
    • 设置缓冲时间

里程碑管理（Milestone Management）

一、学习目标

完成本课后，学习者应能够：

  1. 理解里程碑在项目管理中的作用和价值。
  2. 掌握如何设置、跟踪和管理项目里程碑。
  3. 能够通过里程碑监控项目进度、风险和依赖关系。
  4. 熟悉工具与方法，用于可视化里程碑和阶段成果。

二、核心概念

里程碑（Milestone）是项目中的关键节点，用于标识重大成果或阶段性目标完成。

核心概念

  1. 里程碑的作用

    • 监控项目进度
    • 提供阶段性成果验证
    • 帮助利益相关者了解项目状态
    • 识别潜在延迟和风险

  2. 里程碑类型

    • 项目启动里程碑：项目计划完成、团队组建
    • 阶段完成里程碑：需求冻结、设计完成、开发完成
    • 交付里程碑：内部上线、用户验收、正式发布
    • 审查与评估里程碑：阶段评审、迭代回顾

  3. 里程碑与关键路径

    • 里程碑通常对应关键路径的任务完成点
    • 延迟里程碑会直接影响整体项目交付

范围管理（Scope Management）

一、学习目标

完成本课后，学习者应能够：

  1. 理解项目范围管理的重要性及其对项目成功的影响。
  2. 掌握范围定义、范围控制和变更管理的流程。
  3. 能够制定清晰的项目范围说明书（Project Scope Statement）。
  4. 熟悉工具和方法，用于跟踪范围和处理范围变更。

二、核心概念

项目范围管理（Scope Management）是确保项目仅交付所需成果并控制额外工作的过程。

核心概念

  1. 项目范围（Project Scope）

    • 包含内容（In-Scope）：项目必须交付的产品、服务和成果。
    • 排除内容（Out-of-Scope）：明确项目不涉及的内容，防止范围蔓延。

  2. 范围蔓延（Scope Creep）

    • 项目未经正式变更程序增加或修改需求
    • 会导致成本、时间延长，影响项目质量

  3. 范围管理流程

    1. 范围规划（Scope Planning）
    2. 范围定义（Scope Definition）
    3. 创建工作分解结构（WBS, Work Breakdown Structure）
    4. 范围验证（Scope Verification）
    5. 范围控制（Scope Control）

  4. 范围说明书（Project Scope Statement）

    • 包含项目目标、交付物、可交付标准、排除内容、约束与假设

时间估算（Timeline Estimation）

一、学习目标

完成本课后，学习者应能够：

  1. 理解时间估算在项目管理中的重要性。
  2. 掌握常用的时间估算方法和工具。
  3. 能够为项目任务制定合理的时间计划。
  4. 具备识别时间风险并设置缓冲的能力。

二、核心概念

时间估算是预测完成项目任务所需时间的过程，是项目计划和资源分配的基础。

核心概念

  1. 时间估算类型

    • 粗略估算（Rough Estimate）：项目初期，对整体时间的初步预测
    • 详细估算（Detailed Estimate）：任务级别，考虑依赖关系、资源和风险

  2. 时间估算方法

    • 专家判断（Expert Judgment）：通过有经验人员的经验预测
    • 类比估算（Analogous Estimating）：参考类似项目历史数据
    • 自下而上估算（Bottom-Up Estimating）：从任务拆分逐级累加
    • 三点估算（Three-Point Estimating）：乐观（O）、悲观（P）、最可能（M），计算期望时间 E = (O + 4M + P)/6

  3. 关键路径（Critical Path）与缓冲

    • 关键路径决定项目最短完成时间
    • 为风险和不确定性设置缓冲时间（Buffer）

  4. 甘特图（Gantt Chart）与里程碑

    • 可视化任务、时间、依赖和里程碑
    • 有助于进度跟踪和调整

执行与交付（Execution & Delivery）

一、学习目标

完成本课后，学习者应能够：

  1. 理解项目执行与交付的核心目标与流程。
  2. 掌握任务执行、资源管理和进度监控方法。
  3. 能够确保项目成果按时、按质量和按预算交付。
  4. 熟悉交付验收流程及持续改进方法。

二、核心概念

项目执行与交付是将计划转化为实际成果的过程，确保项目目标得以实现。

核心概念

  1. 项目执行（Project Execution）

    • 按照计划分配资源、执行任务
    • 协调团队成员和外部资源
    • 监控任务进度和质量

  2. 项目交付（Project Delivery）

    • 将成果交付给客户或利益相关者
    • 确认交付物符合要求和标准

  3. 关键成功因素

    • 进度管理：确保任务按时完成
    • 质量管理：遵循验收标准
    • 风险与问题管理：及时应对执行中出现的问题
    • 沟通与协作：保持团队和利益相关者信息同步

  4. 交付验收（Acceptance & Sign-off）

    • 客户或产品负责人验证交付物
    • 记录验收结果和改进建议

架构决策（Architectural Decision-Making）

一、学习目标

完成本课后，学习者应能够：

  1. 理解架构决策在技术领导力和项目成功中的作用。
  2. 掌握架构决策的分析方法和评估标准。
  3. 能够在团队中制定可行的技术方案和决策记录。
  4. 具备平衡性能、可扩展性、成本和风险的能力。

二、核心概念

架构决策是选择系统设计方案和技术栈的过程，直接影响系统的性能、可维护性和可扩展性。

核心概念

  1. 架构决策的目标

    • 确保系统满足业务需求和非功能性需求（性能、可扩展性、安全性）
    • 降低技术债务和未来维护成本
    • 提供团队统一的设计指导

  2. 决策驱动因素

    • 功能需求：系统必须实现的核心功能
    • 非功能需求：性能、可扩展性、可靠性、安全性
    • 技术约束：现有平台、团队技能、预算
    • 风险与不确定性：新技术、外部依赖、时间限制

  3. 架构决策流程

    1. 明确问题和目标
    2. 收集候选方案
    3. 评估方案优缺点
    4. 选择最优方案并记录决策
    5. 持续跟踪和调整

  4. 架构决策记录（ADR, Architectural Decision Record）

    • 记录决策背景、备选方案、决策理由、结果和影响
    • 保持团队共享和历史追踪

系统监控与性能（System Monitoring & Performance）

一、学习目标

完成本课后，学习者应能够：

  1. 理解系统监控与性能管理的重要性。
  2. 掌握关键性能指标（KPI）和监控工具的使用。
  3. 能够设计和实施监控方案，及时发现和解决性能问题。
  4. 学会基于监控数据进行容量规划和优化决策。

二、核心概念

系统监控与性能管理是确保系统稳定运行、满足业务需求的关键工程实践。

核心概念

  1. 关键性能指标（KPI, Key Performance Indicators）

    • 响应时间（Response Time）：用户请求到系统响应的时间
    • 吞吐量（Throughput）：单位时间内系统处理的请求数量
    • 可用性（Availability）：系统正常运行的时间比例
    • 错误率（Error Rate）：系统错误请求占总请求比例
    • 资源利用率（CPU, Memory, Disk, Network）：监控系统负载

  2. 监控类型

    • 基础设施监控（Infrastructure Monitoring）：服务器、网络、存储
    • 应用性能监控（APM, Application Performance Monitoring）：应用内部性能指标
    • 日志监控（Log Monitoring）：错误、警告、访问日志分析
    • 用户体验监控（UX Monitoring）：前端性能和用户操作响应

  3. 性能瓶颈识别

    • CPU 或内存过高
    • 数据库响应慢
    • 网络延迟
    • 应用代码效率低

  4. 容量规划（Capacity Planning）

    • 基于历史数据预测系统扩展需求
    • 提前准备资源以应对流量增长

基础设施扩展（Scaling Infrastructure）

一、学习目标

完成本课后，学习者应能够：

  1. 理解基础设施扩展的必要性及其对系统性能和可用性的影响。
  2. 掌握水平扩展与垂直扩展的策略和场景。
  3. 能够设计可扩展架构以应对业务增长。
  4. 熟悉云服务和自动化扩展工具的使用。

二、核心概念

基础设施扩展是指在系统负载增加时，通过增加或优化资源以维持性能和可用性。

核心概念

  1. 扩展类型

    • 垂直扩展（Vertical Scaling）：增加单台服务器的 CPU、内存或存储能力
    • 水平扩展（Horizontal Scaling）：增加更多服务器实例，实现负载分摊

  2. 可扩展性指标

    • 系统吞吐量（TPS, Transactions Per Second）
    • 响应时间
    • 并发用户数
    • 资源利用率（CPU、内存、网络、存储）

  3. 扩展策略

    • 预扩展（Pre-scaling）：在流量增加前提前部署资源
    • 自动扩展（Auto-scaling）：根据负载动态增加或减少实例
    • 容器化和微服务：提高服务独立部署和扩展能力

  4. 负载均衡（Load Balancing）

    • 分发请求到多实例，防止单点瓶颈
    • 常用策略：轮询、最少连接、基于性能

技术债务管理（Technical Debt Management）

一、学习目标

完成本课后，学习者应能够：

  1. 理解技术债务的概念及其对项目长期健康的影响。
  2. 掌握识别、评估和管理技术债务的方法。
  3. 能够制定减少技术债务的策略，并融入日常开发流程。
  4. 学会平衡交付速度与代码质量，优化系统长期维护性。

二、核心概念

技术债务（Technical Debt）指在软件开发过程中，为了快速交付而做出的权衡，导致后续维护成本增加的问题。

核心概念

  1. 技术债务类型

    • 代码债务：冗余代码、缺乏测试、代码风格不一致
    • 设计债务：架构不合理、模块耦合过高
    • 文档债务：缺乏文档、说明不完整
    • 测试债务：缺少单元测试、自动化测试覆盖不足

  2. 债务成本

    • 利息（Interest）：技术债务导致未来开发、修复成本增加
    • 本金（Principal）：当前系统中存在的技术债务数量

  3. 技术债务管理流程

    1. 识别债务
    2. 评估债务影响和优先级
    3. 制定偿还计划
    4. 实施优化和重构
    5. 持续监控与防止新增债务

技术路线规划（Technical Roadmapping）

一、学习目标

完成本课后，学习者应能够：

  1. 理解技术路线规划在企业技术战略和产品发展中的作用。
  2. 掌握制定技术路线图的方法和步骤。
  3. 能够将业务需求、技术趋势与团队能力结合形成长期规划。
  4. 熟悉工具和方法，实现技术路线可视化与迭代更新。

二、核心概念

技术路线规划（Technical Roadmapping）是对企业或产品未来技术发展方向的战略性规划，用于指导架构、平台和技术决策。

核心概念

  1. 技术路线图（Technical Roadmap）

    • 描述未来技术发展计划、阶段目标和关键里程碑
    • 包括平台升级、架构改进、新技术引入、团队能力建设

  2. 规划原则

    • 对齐业务战略：技术规划支持产品目标和市场战略
    • 可执行性：阶段目标可分解为具体任务
    • 灵活性：可根据业务和技术变化进行调整
    • 透明沟通：技术路线图对团队和利益相关者透明

  3. 路线图层级

    • 战略层：长期技术方向，3~5 年规划
    • 战术层：技术平台、架构演进，1~2 年规划
    • 执行层：迭代、项目计划，季度或月度规划

  4. 关键要素

    • 技术主题（技术升级、架构改进）
    • 时间线和里程碑
    • 风险与依赖关系
    • 资源和预算需求

构建 vs 购买评估（Build vs Buy Evaluation）

一、学习目标

完成本课后，学习者应能够：

  1. 理解“构建 vs 购买”决策在技术选型中的重要性。
  2. 掌握评估自研与采购第三方解决方案的分析方法。
  3. 能够综合成本、时间、风险和长期维护因素做出决策。
  4. 熟悉决策记录和沟通方法，使团队和利益相关者理解选择依据。

二、核心概念

“构建 vs 购买”评估是针对系统或工具选择自研开发或采购现成解决方案的决策过程。

核心概念

  1. 决策驱动因素

    • 成本（Cost）：开发成本 vs 采购许可和维护费用
    • 时间（Time-to-Market）：交付速度需求
    • 定制化（Customization）：业务需求的特殊性
    • 可维护性（Maintainability）：长期运维和升级难度
    • 风险（Risk）：技术不成熟、供应商依赖、合同风险

  2. 评估方法

    • 加权评分法（Weighted Scoring）：根据各项因素设定权重，评分决定最佳选项
    • ROI 分析（Return on Investment）：对比成本与潜在收益
    • 总拥有成本（TCO, Total Cost of Ownership）：考虑长期维护和运营成本

  3. 决策流程

    1. 定义需求和目标
    2. 收集自研和第三方方案信息
    3. 对比成本、时间、风险、可维护性
    4. 做出决策并记录理由
    5. 定期复审和调整

技术风险评估（Technical Risk Assessment）

一、学习目标

完成本课后，学习者应能够：

  1. 理解技术风险在项目管理中的重要性。
  2. 掌握识别、评估和缓解技术风险的方法。
  3. 能够制定风险管理计划，并将其纳入项目执行。
  4. 熟悉工具和流程，实现持续监控和反馈。

二、核心概念

技术风险评估是识别项目中可能影响系统性能、可用性或交付质量的技术问题，并制定应对策略的过程。

核心概念

  1. 技术风险类型

    • 架构风险：设计不合理、可扩展性不足
    • 代码风险：技术债务高、缺乏测试覆盖
    • 基础设施风险：硬件、云服务或网络故障
    • 集成风险：依赖第三方服务或组件不可用
    • 安全风险：漏洞、数据泄露或合规问题

  2. 风险评估指标

    • 概率（Probability）：风险发生的可能性
    • 影响（Impact）：风险发生对项目或业务的影响
    • 优先级（Priority）：结合概率和影响，确定处理顺序

  3. 风险管理流程

    1. 风险识别（Risk Identification）
    2. 风险分析与评估（Risk Analysis & Assessment）
    3. 风险缓解计划（Risk Mitigation Plan）
    4. 风险监控与复审（Risk Monitoring & Review）

安全与测试策略（Security & Testing Strategies）

一、学习目标

完成本课后，学习者应能够：

  1. 理解安全与测试在软件工程中的核心作用。
  2. 掌握常用测试策略、测试类型和安全防护措施。
  3. 能够设计测试计划和安全策略以保障系统质量。
  4. 熟悉集成安全和测试实践于 CI/CD 流程的方法。

二、核心概念

安全与测试策略确保软件系统在功能、性能和安全性方面满足预期，降低风险。

核心概念

  1. 测试策略（Testing Strategies）

    • 单元测试（Unit Testing）：验证单个函数或模块
    • 集成测试（Integration Testing）：验证模块间交互
    • 端到端测试（End-to-End Testing）：验证完整业务流程
    • 回归测试（Regression Testing）：确保改动未破坏原有功能
    • 自动化测试（Automation Testing）：提高测试效率和覆盖率

  2. 安全策略（Security Strategies）

    • 身份认证与授权（Authentication & Authorization）
    • 数据加密（Encryption）：传输和存储安全
    • 漏洞扫描与渗透测试（Vulnerability Scanning & Pen Testing）
    • 安全编码规范（Secure Coding Practices）
    • 日志与监控（Logging & Monitoring）

  3. 测试与安全在开发流程中的整合

    • Shift-Left Testing：尽早在开发阶段进行测试
    • DevSecOps：安全嵌入 CI/CD 流程
    • 持续监控与反馈：实时发现问题并修复

招聘与团队结构（Hiring & Team Design）

一、学习目标

完成本课后，学习者应能够：

  1. 理解招聘与团队结构对工程团队绩效的重要性。
  2. 掌握招聘流程、面试方法及评估候选人技能的技巧。
  3. 能够设计合理的团队结构，优化团队协作与效率。
  4. 熟悉团队扩张和角色分配策略，支持长期发展。

二、核心概念

招聘与团队设计是构建高效、可持续工程团队的基础，涉及人才获取、角色分配和团队协作机制。

核心概念

  1. 招聘流程

    • 需求分析：明确岗位职责和技能要求
    • 职位发布：选择合适渠道吸引候选人
    • 筛选简历：匹配技能和经验
    • 面试评估：技术能力、文化契合度、软技能
    • 录用与入职：提供 offer 并安排入职培训

  2. 团队结构设计

    • 职能型（Functional Team）：按专业技能分组
    • 项目型（Project Team）：围绕项目目标组建跨职能团队
    • 矩阵型（Matrix Team）：结合职能和项目目标，灵活分配资源
    • 自组织型（Self-Organizing Team）：团队自主分配任务，强调协作与创新

  3. 关键角色

    • 工程经理 / 技术主管
    • 开发工程师（前端、后端、全栈）
    • 测试工程师 / QA
    • DevOps / 系统管理员
    • 产品经理 / 设计师（跨职能团队）

  4. 招聘评估指标

    • 技术能力
    • 问题解决能力
    • 学习能力与成长潜力
    • 团队协作与沟通能力
    • 文化契合度
